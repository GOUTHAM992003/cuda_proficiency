# 1.1 Heterogeneous Parallel Computing

Since 2003, the semiconductor industry hit a physical wall with single-core processors and split into two distinct paths for designing chips: the **Multicore trajectory (CPUs)** and the **Many-thread trajectory (GPUs)**. 

Because of their vastly different design philosophies, there is now a massive performance gap between the two. This gap has motivated modern applications (like Deep Learning) to use a "heterogeneous" approach—using both chips together to get the best of both worlds.

Let's break down exactly how these two paths differ in depth, step by step.

---

## 1. The Multicore Trajectory: CPUs (Latency-Oriented)

The CPU path is all about maintaining the execution speed of **sequential programs** (tasks that must happen one after another, in order). 

*   **The Goal:** Finish a single task (or thread) as absolutely fast as possible. This is called a **latency-oriented design**.
*   **How it works:** CPUs use very complex hardware to minimize the time it takes to do an operation. They have large memory caches to avoid waiting for data, and highly sophisticated "Control" logic (like branch prediction) to guess what the code will do next so the CPU never stalls.
*   **The Cost:** Reducing latency is incredibly expensive in terms of hardware. For example, if you want to cut the time of a math operation in half, it might require *quadrupling* the power consumption and more than doubling the chip area. Because of this, CPUs can only fit a small number of cores on a chip.
*   **Real-World Example & Metrics:** A recent Intel multicore server CPU has up to 24 cores. Its peak calculation speed is **0.33 TFLOPS** (TeraFLOPS) for double-precision math, and **0.66 TFLOPS** for single-precision math.

## 2. The Many-Thread Trajectory: GPUs (Throughput-Oriented)

The GPU path was originally driven by the massive economic pressure of the video game industry, which needed huge amounts of simple math to render millions of pixels per frame. 

*   **The Goal:** It doesn't care how long one individual task takes. It cares about the total amount of work finished over a period of time. This is called a **throughput-oriented design**.
*   **How it works:** Instead of spending chip area on complex Control logic and massive caches, GPUs rip all that out. They use that saved chip area and power budget to pack tens of thousands of simple, tiny "Workers" (ALUs) and pipelined memory channels onto the chip. 
*   **Memory Bandwidth:** Because graphics require moving massive amounts of data in and out of DRAM rapidly, GPUs are designed with roughly **10 times the memory bandwidth** of contemporaneously available CPU chips. 
*   **Real-World Example & Metrics:** The NVIDIA A100 GPU (Ampere architecture) absolutely crushes the CPU in raw speed. It achieves **9.7 TFLOPS** for double-precision (vs Intel's 0.33), **156 TFLOPS** for single-precision (vs Intel's 0.66), and a staggering **312 TFLOPS** for half-precision. *(Note: The massive 156 and 312 TFLOPS figures specifically represent the A100's specialized **Tensor Core** performance, which accelerates deep learning math, rather than standard operations).*

## 3. Visualizing the Architecture Differences

To truly understand why the A100 GPU is hundreds of times faster at raw math than the 24-core Intel CPU, we have to look at how their physical silicon is divided up.

![CPU vs GPU Architecture](./1.Introduction%20/fig_1.1_cpu_vs_gpu_architecture_v2.png)
*Figure 1.1: CPUs and GPUs have fundamentally different design philosophies: (A) CPU design is latency-oriented; (B) GPU design is throughput-oriented.*

### Breaking Down the Diagram:
1.  **Control (Yellow):** 
    *   *CPU:* Huge. Packed with complex logic to predict code branching and manage a single thread perfectly.
    *   *GPU:* Tiny. It doesn't predict anything. It just feeds simple instructions to thousands of workers.
2.  **Cache (Orange):** 
    *   *CPU:* Massive. It stores lots of data close to the workers so they never have to wait (minimizing latency).
    *   *GPU:* Very small. It only exists to help control bandwidth so multiple workers accessing the same data don't overwhelm the main memory.
3.  **ALU / Workers (Green):** 
    *   *CPU:* Only a few, but they are very large and complex (e.g., 4 in the diagram).
    *   *GPU:* A massive sea of hundreds of tiny, simple arithmetic workers. 
4.  **DRAM / Main Memory (Bottom Strip):** The GPU has many more memory channels to feed its thousands of workers, resulting in its ~10x bandwidth advantage.

**How the GPU hides latency without a big Cache:**
If a GPU thread needs data from the slow main memory (DRAM), it takes a long time. But the GPU doesn't care! Because it is managing thousands of parallel threads, when one group of threads is stuck waiting for memory, the GPU hardware instantly swaps to another group of threads that are ready to do math. It constantly finds work to do, keeping total throughput sky-high.

## 4. What is Heterogeneous Computing?

Because of these fundamental differences, CPUs and GPUs are good at completely different things. 
*   **CPUs** are designed to excel at **complex, sequential tasks** (like running an operating system or game logic) that use very few threads. 
*   **GPUs** are designed to excel at **massive, highly-parallel tasks** (like rendering graphics or deep learning math) that use thousands of threads.

**Heterogeneous Computing** is the practice of using them both together in the same system. An application runs its sequential parts on the CPU, and its numerically intensive parts on the GPU. 

### Why did GPUs become the standard for Parallel Computing?
In the early days, if you wanted to do massive parallel computing, you had to buy gigantic, expensive data-center cluster servers. GPUs changed the world for three main reasons:
1.  **Installed Base (Economics):** Because of the PC gaming market, GPUs were already being sold by the hundreds of millions. Today, there are over **1 billion CUDA-enabled GPUs** in use. This massive customer base made software development economically viable.
2.  **Form Factor & Accessibility:** Instead of needing a massive server rack, you could just plug a GPU into a normal desktop PC. For example, clinical MRI machines in hospitals couldn't use giant server racks, but they could easily fit a PC with a GPU inside. Today, many companies ship MRI products powered by GPUs.
3.  **The Invention of CUDA (2007):** Before 2007, GPUs were terribly difficult to use for general math. Programmers had to "trick" the graphics API (OpenGL or Direct3D) by writing their math as if it was painting pixels. This was called **GPGPU** (General Purpose GPU). It was hard, and not widespread. 

Everything changed in 2007 when NVIDIA released **CUDA**. They literally added physical hardware to the chip to bypass the graphics interface, creating a general-purpose programming interface right on the silicon. They redid the software layers so developers could use familiar C/C++ tools. This opened the floodgates for easily developing applications for GPUs, making heterogeneous computing the powerhouse it is today.
