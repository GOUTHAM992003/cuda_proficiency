# 1.3 Speeding Up Real Applications

When we write parallel code for a GPU, the ultimate goal is to make the program run faster. But exactly how much faster will it get? And what are the hard limits holding us back?

To understand this, we need to look at how we calculate speed, the mathematical limits of parallel programming (Amdahl's Law), the optimistic alternative (Gustafson's Law), and the fundamental trade-offs between Latency and Throughput.

---

## 1. What is "Speedup"?

Before we do any math, we need to define what **speedup** actually means in a very simple way.

Speedup is just the ratio of the time an application takes on an old system compared to a new system.

* **Formula:** `Speedup = (Time on Old System) / (Time on New System)`

**Example:**
If your application takes 200 seconds to run on your old standard CPU (System B), and only 10 seconds to run on a shiny new GPU (System A):

* Speedup = 200 / 10 = **20x speedup** (meaning your code is now 20 times faster).

---

## 2. Amdahl's Law (The Time-Based Speedup)

You might think that if you buy a GPU that is 100x faster than your CPU, your entire application will automatically finish 100x faster. **This is completely false.**

Amdahl's Law measures how much faster a *fixed, original task* becomes. It states that the overall speedup of your application is strictly limited by the portion of your code that *cannot* be run in parallel (the serial code).

Every application is split into two parts:

1. **Serial/Sequential Portion ($T_s$):** Code that must run one step at a time.
2. **Parallel Portion ($T_p$):** Math-heavy code where calculations can happen simultaneously.

### The Math of Amdahl's Law

Amdahl's speedup ($S_a$) focuses on the **ratio of time**.
$$
S_a = \frac{T_s + T_p}{T_s + \frac{T_p}{n}}
$$
*(Where $n$ is the number of cores/threads)*

Let's say a program has 10% serial code ($T_s = 0.1$) and 90% parallel code ($T_p = 0.9$), and we run it on 28 threads ($n = 28$).

* $S_a = \frac{1}{0.1 + (0.9 / 28)} \approx 7.5x \text{ faster}$

Even with 28 threads, the absolute maximum speedup we got was 7.5x. The 10% serial part dominates the timeline as the parallel part shrinks. This represents a hard limit.

### The Gamer's Case-Study (Amdahl's Realm)

In a video game, the problem size is **fixed**. You need to calculate one specific frame.
If the game's "Physics Engine" is serial, it does not matter if you have 28 threads or 10,000 threads. The frame *cannot* be finished until that one serial thread is done crunching the physics.

This is exactly why PC gamers prefer CPUs like the Intel i7-14700K over massive 64-core Server CPUs. The i7-14700K has **Performance Cores (P-Cores)** with extremely high clock speeds to finish that serial part ($T_s$) as fast as physically possible.

---

## 3. Gustafson's Law (The Work-Based Speedup)

If Amdahl's Law feels pessimistic, it is because it assumes a fixed workload. John Gustafson argued that when we buy massive parallel hardware, we don't just solve tiny problems faster—we scale up the problem size!

Gustafson's Law measures **how much MORE work is done in the same amount of time.**

### The Math of Gustafson's Law

If we take Amdahl's formula and rearrange it to solve for the *amount of work done* in a specific time, we get Gustafson's speedup ($S_g$):
$$
S_g = T_s + n(T_p)
$$

Let's use the exact same scenario: 10% serial ($T_s = 0.1$), 90% parallel ($T_p = 0.9$), on 28 threads ($n = 28$).

* $S_g = 0.1 + 28(0.9) = 25.3x \text{ more work}$

**Why the Gap? (7.5x vs 25.3x)**
In Amdahl's world, the 10% serial part dominates the timeline as the parallel part shrinks. In Gustafson's world, you added so much extra parallel work that the 10% serial part becomes a tiny speck compared to the massive mountain of work you finished.

### The Data Scientist Case-Study (Gustafson's Realm)

Imagine a data scientist processing 100GB of data. On 1 core, it takes 100 hours.

* **Using Amdahl's view:** They use 28 threads to try and finish that 100GB in 4 hours. But the serial setup time dominates, bottlenecking the speed.
* **Using Gustafson's view:** They realize they have 28 threads, so instead of just processing 100GB faster, they process **2.8 Terabytes** in the same 100 hours!

Because the "setup/serial" time to start the program is often the same whether you process 1GB or 1TB, the efficiency of the 28 threads is massive when the workload scales up.

---

## 4. Extreme Cases: Are Both Laws the Same Reality?

Yes! Both laws mathematically conclude that **serial code is the enemy**.

1. **If $T_s = 1.0$ (100% Serial Code):**
    * Amdahl Speedup = $\frac{1}{1 + 0} = 1$ (No speedup)
    * Gustafson Speedup = $1 + 28(0) = 1$ (No extra work done)
2. **If $T_s = 0.0$ (100% Parallel Code):**
    * Amdahl Speedup = $\frac{1}{0 + \frac{1}{n}} = n$
    * Gustafson Speedup = $0 + n(1) = n$

**The Core Takeaway:**

* **Amdahl's Law** tells you the limit of speed for a *specific problem you already have*.
* **Gustafson's Law** tells you the potential of your hardware to handle *much bigger problems than you currently have*.

## 5. How to Determine Your Parallel Percentage in Reality?

How do you actually know if your code is 90% parallel or 50% parallel? You find out by **profiling the code**.

In Linux, we can use tools like `perf`. It will tell  exactly which functions are taking up time, and whether they are scaling beautifully across multiple cores or getting stuck waiting on a single thread.

**Reverse-Engineering Amdahl's Law:**
You can actually calculate your code's exact parallel fraction ($P$) by measuring its execution time on 1 thread ($T_1$) and then on $N$ threads ($T_n$) to find the real-world speedup. Once you have the speedup, use this reversed formula:
$$
P = \frac{n \times (\text{Speedup} - 1)}{\text{Speedup} \times (n - 1)}
$$

**The Proof (Deriving $P$ from Amdahl's Law):**
Let $S$ be Speedup, $P$ be the parallel fraction, and $n$ be the number of threads.

1. Start with Amdahl's Law: $S = \frac{1}{(1-P) + \frac{P}{n}}$
2. Take the reciprocal of both sides: $\frac{1}{S} = (1-P) + \frac{P}{n}$
3. Rearrange to isolate the $P$ terms: $\frac{1}{S} - 1 = -P + \frac{P}{n}$
4. Factor out $P$: $\frac{1 - S}{S} = P \left( \frac{1}{n} - 1 \right)$
5. Find a common denominator inside the parenthesis: $\frac{1 - S}{S} = P \left( \frac{1 - n}{n} \right)$
6. Multiply both sides by $-1$ to flip the signs: $\frac{S - 1}{S} = P \left( \frac{n - 1}{n} \right)$
7. Isolate $P$ by multiplying both sides by $\frac{n}{n - 1}$: $P = \frac{n \times (S - 1)}{S \times (n - 1)}$

## 6. Latency vs. Throughput (The Cost of Speed)

To understand why CPUs and GPUs are built differently, we must look at the two different ways to make code run faster.

| Feature | Increasing Throughput (GPU Style) | Reducing Latency (CPU Style) |
| :--- | :--- | :--- |
| **Strategy** | Add more "Chefs" (Cores) | Make one "Chef" move faster |
| **Chip Area** | Scales Linearly (Add 1 core, get 1 core) | Scales Poorly (Double the area, only get ~1.5x speed) |
| **Power Cost** | Linear (Double the cores = Double power) | Exponential (Cutting time in half requires quadruple power) |
| **Philosophy**| Optimize for *many* threads | Optimize for *one* fast thread |

### The Expensive Cost of Latency

**Reducing latency is much more expensive than increasing throughput.**
If you want to double your throughput (math speed), you just "copy and paste" a second Arithmetic Logic Unit (ALU) onto the chip. It's a 1:1 trade—double the area and power, double the work.

Making a *single* core finish calculations faster requires incredibly complex, power-hungry engineering. This explains why CPU cores are so much bigger and hotter than GPU cores.

### The i7-14700K Hybrid Architecture

Modern processors like the Intel i7-14700K elegantly combine both philosophies:

* **8 P-Cores (Performance):** These are Latency cores. They use a massive amount of area and power to finish your serial code ($1-P$) as fast as physically possible. They are the power-hungry monsters designed to crush the serial bottleneck.
* **12 E-Cores (Efficient):** These are Throughput cores. They are small and efficient, allowing Intel to pack 12 of them into a tiny space. When your code reaches the parallelizable portion ($P$), you want as many hands as possible. They handle the massive parallel workloads without melting the chip.

## 7. Clock Speed, IPC, and The Conveyor Belt Analogy

You often hear CPU speeds measured in Gigahertz (GHz). But what does a "Hertz" actually mean?

Inside the CPU, there is an internal drummer (a quartz crystal oscillator). It sends out a steady pulse of electricity—a square wave—to the entire chip. Nothing happens without a clock pulse; it is the "GO!" signal for transistors to fetch, decode, execute, and store instructions.

* $1 \text{ Hertz (Hz)} = 1 \text{ pulse per second}$
* $5.6 \text{ GHz} = 5.6 \text{ billion pulses (ticks) every single second}$

**(Note: Because computer architecture cares about time, we look at the Cycle Time ($T$). For a 5.6 GHz core, $T = \frac{1}{5,600,000,000} \approx 0.178 \text{ nanoseconds per cycle}$).**

### Does 5.6 GHz mean 5.6 Billion Instructions per second?

**No.** It means 5.6 billion *clock cycles* per second. The actual number of instructions depends on **IPC (Instructions Per Cycle)**. Modern processors are *superscalar*, meaning they have multiple execution units inside a single core. If a P-Core has an IPC of 5, it is actually retiring $5.6 \times 5 = 28 \text{ billion}$ instructions per second!

### The Factory Conveyor Belt Analogy (Why Stalls Matter)

Why don't we just measure "Instructions Per Second"? Because **not all instructions are equal**.

Think of 5.6 GHz as the speed of a conveyor belt in a factory:

* **The Clock (5.6 GHz):** The belt moves exactly 5.6 billion times a second.
* **The Instructions:** The products being placed on the belt.
* **High IPC:** If the workers (transistors) are fast, they place a product on every slot of the belt (e.g., simple `ADD 1+1` math).
* **Memory Stalls:** If the factory runs out of parts (Memory Latency/Waiting for RAM), the belt *still moves* at 5.6 GHz, but the slots are completely empty (Zero Instructions passing through).

If your code is stalled waiting for slow main memory, having a 5.6 GHz clock speed doesn't help you at all.

## 8. The "Peach" Analogy (Figure 1.2)

To visualize how applications are divided between CPUs and GPUs, the textbook uses the famous "Peach" analogy.

![The Anatomy of a Superapplication](./1.Introduction%20/fig_1.2_peach_anatomy.png)
*Figure 1.2: Coverage of sequential and parallel application portions. The peach pit represents the sequential code, and the flesh represents the parallel code.*

### Breaking Down the Picture

1. **The Pit (Dark Brown Center):**
    This represents the **Sequential Code**. It is hard, tough, and impossible to break apart into parallel threads. Trying to run this on a GPU is "like biting into a peach pit —>not a good feeling!" The CPU is designed to crunch through this pit very well. While the pit might take up a lot of the actual lines of code you wrote in your file, it usually only takes up a small portion of the total execution *time* in a massive application.
2. **The Flesh (Orange Area):**
    This represents the **Data Parallel Code**. It is huge, sweet, and easy to chop up into thousands of pieces. This is where the GPU thrives and where the application spends almost all of its processing time.
3. **Traditional CPU Coverage (Light Blue Grid):**
    Notice how the CPU grid perfectly covers the pit, but only covers a tiny bit of the orange flesh. The red "Obstacle" lines pushing back show the physical power constraints that make it impossible for a single-core CPU to eat the rest of the massive parallel flesh. It just isn't built for it.
4. **GPGPU Coverage (Dark Blue Patch):**
    Before CUDA existed, people used "GPGPU" (tricking graphics cards into doing math by disguising the math as pixel-painting). As you can see, this only covered a very tiny sliver of the peach flesh because it was severely limited to a specific type of math.
5. **The Goal of CUDA:**
    The entire point of the modern CUDA programming model is to act as a massive mouth that can eat a much larger chunk of that orange peach flesh, allowing us to effortlessly parallelize massive sections of our applications.

## 9. Real-World Hardware: Inside Your RTX 3060 (Ampere)

Now that we understand the difference between CPU latency cores and GPU throughput cores, let's look at the exact hardware you are using: the **NVIDIA GeForce RTX 3060**.

Your RTX 3060 is built on the **Ampere Architecture** using the **GA106** silicon chip. Let's break down its specs exactly like we did for the Intel CPU:

* **SMs (Streaming Multiprocessors):** 28
* **L2 Cache:** ~3 MiB
* **CUDA Cores per SM:** 128
* **Total CUDA Cores:** $28 \times 128 = \mathbf{3,584}$
* **Tensor Cores per SM:** 4
* **Total Tensor Cores:** $28 \times 4 = \mathbf{112}$ (These are specialized units for Matrix math/AI).

### What Actually is a "Thread"?

Before we do the math on how many threads this GPU handles, we need to clarify what a thread is.

**A thread is a software abstraction, not a physical hardware wire.**
A thread is simply a sequential stream of execution.
It consists of :

1. **A Program Counter (PC):** A "bookmark" that keeps track of exactly which line of code this specific thread is currently reading.
2. **A Set of Registers:** A tiny, ultra-fast memory bank that holds the local variables and math results this thread is currently calculating.
3. **Execution State:** The current context of the thread.

When we talk about a GPU "managing" threads, we are really talking about how many of these "bookmarks and register states" the hardware can physically keep track of at the exact same time.

### "Managed" Threads vs. "Executed" Threads (Per Cycle)

My Ampere SMs group threads into bundles of 32, called a **Warp**.

* One Ampere SM can manage a maximum of **48 Warps** at the same time.
* $48 \text{ warps} \times 32 \text{ threads} = \mathbf{1,536 \text{ threads per SM}}$.
* Total Threads across the whole GPU: $28 \text{ SMs} \times 1,536 = \mathbf{43,008 \text{ concurrent threads}}$.

I noted that $28 \times 1536 = 43,008$. But what does this massive number actually mean? Does the GPU do 43,008 math equations in a single clock cycle?

**No. Here is where the magic of Latency Hiding comes in.**

1. **Executing per Clock Cycle:**
    Inside a single SM, there are only **4 Warp Schedulers**. This means at any exact, single *clock cycle* (tick), the SM can only physically pick 4 warps to do math.
    * $4 \text{ warps} \times 32 \text{ threads} = \mathbf{128 \text{ threads executing per SM}}$.
    * Across the whole GPU: $28 \text{ SMs} \times 128 = \mathbf{3,584 \text{ threads executing per cycle}}$.
    *(Notice how 3,584 is the exact number of CUDA Cores! A "CUDA Core" is just the physical ALU doing the math).*

2. **Managed per Clock Cycle (In-Flight):**
    If only 3,584 threads can run per cycle, why does the GPU need to hold 43,008 threads?
    Remember the "Conveyor Belt" analogy and memory stalls? If the 3,584 threads currently doing math suddenly need to pull data from slow RAM, they **stall**. Instead of letting the 3,584 CUDA cores sit idle and waste clock cycles, the 4 Warp Schedulers instantly swap out the stalled warps and pick a new set of warps from the remaining pool of 43,008 threads.

Because all 43,008 threads have their Program Counters and Registers physically saved in the SM's hardware, this swap takes **Zero Clock Cycles**. This is how a GPU achieves massive throughput: it always has backup work ready to go when a thread stalls.

### The Register File: How Many Registers per Thread?

To manage 1,536 threads at once, the SM needs a massive pool of registers.
An Ampere SM contains a physical hardware block called the **Register File**, which holds **65,536 (64K) 32-bit registers**.

* **The "Maximum Occupancy" Scenario (~42 Registers):**
    If I want my SM to run at its absolute maximum limit of 1,536 threads (to get the best Latency Hiding), then the 64K Register File must be divided evenly.
    $65,536 \div 1,536 \approx \mathbf{42 \text{ registers per thread}}$.
    If my code uses 42 registers or less, the GPU can safely manage the max 1,536 threads.

* **The "Complex Math" Scenario (Up to 255 Registers):**
    What if I write an extremely complex kernel with huge equations and lots of local variables? My compiler might decide that a single thread needs a massive amount of space to hold its math.
    **The physical hardware limit in the Ampere architecture is exactly 255 registers per thread.** A thread cannot request 256.

    But there is a **Trade-off:**
    If a thread demands 255 registers, the SM physically *cannot* fit 1,536 threads anymore. The math changes:
    $65,536 \div 255 \approx \mathbf{257 \text{ threads}}$.
    Because threads must be launched in Warps (groups of 32), the SM will throttle itself and cap out at managing exactly **256 threads** (8 warps) instead of 1,536.

### The "Restaurant Analogy": Mapping Threads, Registers, and Blocks

To understand how $3,584$ executing threads, $43,008$ managed threads, $1024$ threads per block, and up to $255$ registers all map together, I like to think of a single SM as a **Restaurant**.

1. **The Chefs (Warp Schedulers - Execution Limit):**
    Inside my restaurant (SM), there are 4 chefs (Warp Schedulers) in the kitchen. In one exact second (1 clock cycle), they can only physically chop vegetables for 4 orders at the same time. Since 1 order = 1 warp (32 threads), the chefs can physically "execute" math for **128 threads** per clock cycle. 
    Across the entire 28-SM GPU, that means exactly $28 \times 128 = \mathbf{3,584 \text{ threads executing per cycle}}$. *(This perfectly matches the number of physical CUDA Cores!).*

2. **The Seats (Managed Threads - Capacity Limit):**
    Even though the chefs can only cook 128 meals at once, the restaurant has enough physical floor space to seat **1,536 customers** (threads) at the same time. 
    Across the whole GPU, that is $28 \times 1,536 = \mathbf{43,008 \text{ concurrent threads}}$. 
    
    *Why have 1,536 seats if the chefs can only cook for 128 at a time?*
    **Latency Hiding!** If the 128 people currently eating suddenly ask for a glass of water (a Memory Stall, waiting for RAM), the chefs do not just stand there doing nothing. Because there are 1,408 other people already seated in the restaurant (their states are safely saved in the hardware), the chefs instantly pivot and start cooking for a different table. If the restaurant only had 128 seats, the chefs would just freeze and waste time every time someone asked for water.

3. **The Party Size (Blocks):**
    Customers don't walk into the restaurant one-by-one; they arrive in groups called **Blocks**. I configure these blocks in my C++ code (e.g., `<<<grid, block>>>`). 
    A block can have a maximum of 1024 threads. If a party of 1024 walks in, the host seats them. The restaurant has 1,536 total seats, so there are $1536 - 1024 = 512$ seats left. The host can then seat a second party (block) of 512 threads. The Block is just how the threads are grouped when they enter the SM!

4. **The Size of the Tables (Registers):**
    This is the most critical part: **Who decides how many registers a thread gets?**
    *The Compiler (NVCC)* decides this when I build my code. It looks at the math inside my kernel. If my kernel has 5 variables, NVCC assigns 5 registers per thread. If I write crazy machine-learning math with 100 variables, NVCC assigns 100 registers per thread.
    
    The restaurant has a fixed amount of physical table wood (the **64K Register File**). 
    *   **Maximum Occupancy (~42 Registers):** If each customer only needs a tiny 1-person table (up to 42 registers), the host can easily fit the maximum 1,536 seats into the restaurant. 
    *   **Complex Math (Up to 255 Registers):** If I write incredibly complex code and NVCC assigns the absolute physical limit of 255 registers per thread, each customer now demands a massive 6-person banquet table all to themselves! Because the physical floor space of the restaurant is fixed, the host is forced to remove seats. The restaurant can now only seat **256 people total** instead of 1,536.

### How Do I Decide Which Is Better?

When writing code, I have to make an architectural choice between **Latency Hiding (Occupancy)** vs. **Complex Math (Instruction-Level Parallelism)**.

*   **When to use FEWER registers (High Occupancy):** If my code does a lot of memory reads (fetching from slow RAM), I desperately need high occupancy (1,536 seats). This ensures the chefs always have backup tables to cook for while the other tables wait for RAM. I can actually *force* the compiler to limit registers by passing a flag like `-maxrregcount=32` when compiling!
*   **When to use MORE registers (Low Occupancy):** If my code does intense, localized math that never needs to ask for data from slow RAM, I don't care about latency hiding. I want massive tables (more registers) so the thread can calculate complex math at blazing speed without ever spilling data out to slow memory.

### What does the `-arch=sm_86` flag mean?

When you compile your CUDA code, you use a flag like `-arch=sm_86`.

* `sm` stands for **Streaming Multiprocessor** architecture.
* `86` stands for **Compute Capability 8.6**.

NVIDIA groups their architectures by generations (Compute Capabilities).

* The major number **8** means this is the **Ampere** generation (Turing was 7, Volta was 7.0, Pascal was 6).
* The minor number **6** means it is a specific variant of Ampere (the GA10x series used in GeForce RTX 30-series gaming cards). For comparison, the massive A100 datacenter GPU is Compute Capability `8.0`.

When you tell the compiler `arch=sm_86`, you are saying: *"Hey NVCC, I am going to run this code on an Ampere gaming GPU. Please translate my C++ code into machine instructions that perfectly fit 4 Warp Schedulers, a 64K Register File, and Gen-2 Tensor Cores."* This ensures your binary code is optimized for the exact physical layout of your RTX 3060.
